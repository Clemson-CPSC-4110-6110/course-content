Introduction

Why Git for Unity is Different

Unity projects bring unique challenges compared to typical programming projects:

Binary Files

Textures, audio clips, 3D models — Git wasn't originally designed for these

File Size

Assets can be tens or hundreds of megabytes. GitHub has file size limits.

Starter projects are already configured to handle both challenges!

Accepting & Cloning Assignments

GitHub Classroom Workflow

  1. Click the GitHub Classroom link from Canvas
  2. Select your name from the roster (first time only)
  3. Accept the assignment
  4. GitHub creates your personal copy of the repository

Group Projects: Join an existing team or create a new one — the whole team shares a single repository

Cloning with GitHub Desktop

  1. On GitHub, click the green Code button
  2. Select Open with GitHub Desktop
  3. Choose where to save the project
  4. Click Clone

Windows Users

Put your project folder directly on C:\ drive. Unity creates deeply nested folders that can cause path length issues.

Command Line Alternative

git clone https://github.com/...
  • Copy the HTTPS URL from the Code button
  • Use SSH URL only if you've set up SSH keys
  • GitHub Desktop is perfectly fine for this course

Understanding Project Configuration

The .gitignore File

Tells Git which files and folders to ignore (not track or upload)

Key Exclusions

  • Library/ — Cached data, regenerated automatically
  • Temp/ & Logs/ — Temporary runtime files
  • Build/ — Compiled application output
  • .vs/ & .idea/ — IDE settings

Don't modify!

Don't delete this file or remove entries unless you really know what you're doing

The Library Folder

The most important thing to exclude

  • Stores cached data, compiled scripts, processed assets
  • Can easily be several gigabytes
  • Unity regenerates it automatically when you open the project
  • No reason to upload — wastes space and time

The .gitattributes File

Configures how Git handles different file types

Git LFS

Configures Large File Storage for binary files (textures, audio, 3D models)

Line Endings

Normalizes line endings so files work across Windows and Mac

Why Git LFS?

  • Regular Git stores complete history of every file
  • Text files: Git stores just the differences (efficient)
  • Binary files: Git stores complete copies (expensive!)

Git LFS Solution

Large binary files stored separately on GitHub's servers. Your repository contains small pointer files. Everything "just works" — add, commit, push, pull as normal.

The .meta Files

Every asset and folder has a corresponding .meta file

  • Store import settings and unique identifiers
  • Unity uses these to track assets and their relationships
  • Deleting a .meta file can break script-to-GameObject connections

Never add .meta files to .gitignore!

They must be in version control

The Daily Workflow

Workflow Overview

  1. Pull — Get latest changes
  2. Work — Make changes in Unity
  3. Review — Check changed files
  4. Commit — Save locally with message
  5. Push — Upload to GitHub

Step 1: Pull First

The first thing you should do when sitting down to work

  • In GitHub Desktop: Click Fetch origin
  • If changes exist, click Pull
  • Especially important for group projects

Skipping this step can cause merge conflicts later!

Step 2: Make Changes

Work on your Unity project normally:

  • Edit scenes
  • Write code
  • Import assets
  • Adjust settings

Nothing special here — just do your work!

Step 3: Review Changes

In GitHub Desktop, check the Changes tab

Normal

  • Scene files you edited
  • Scripts you modified
  • Associated .meta files

Warning Signs

  • Hundreds of unexpected files
  • Hundreds of MB of changes
  • Files that should be ignored

If something looks wrong, ask before committing!

Step 4: Write Commit Messages

A short description of what you changed and why

Good Messages

  • "Added player movement script with jump functionality"
  • "Fixed collision detection bug in enemy AI"
  • "Updated main menu UI layout"

Bad Messages

  • "Updated stuff"
  • "Changes"
  • "asdf"

Step 5: Push to GitHub

  • Committing saves changes locally
  • Pushing uploads them to GitHub
  • Click Push origin in GitHub Desktop

Push Frequently!

  • At least at the end of every work session
  • Ideally after every logical chunk of work
  • Small, frequent pushes are easier to manage

Being a Good Steward of Storage

Why Storage Matters

  • Our class shares a storage pool on GitHub
  • Git LFS has limits on storage space and bandwidth
  • Bandwidth is used every time someone clones or pulls
  • Being wasteful can block you or teammates from pushing

What NOT to Commit

Build Outputs

Often hundreds of MB. Can be regenerated from source. Use file sharing if needed.

Large Asset Store Packages

Document which packages you use in a README instead. Each person downloads them.

WIP Large Assets

Keep iterating textures/audio local until finalized. Every version takes space.

What TO Commit

Everything that actually makes up your project:

  • Scripts
  • Scenes
  • Prefabs
  • Final assets
  • ProjectSettings folder
  • Package manifest
  • All .meta files

Rule of thumb: If someone cloned your repo, could they open the project and have it work?

Quick Sanity Check

Before every commit, ask yourself:

  • Did I only edit a script? I should see that script and maybe its .meta file.
  • Am I seeing hundreds of MB of changes? Pause and investigate.
  • Is the .gitignore working correctly?
  • Am I about to commit a build folder?

When in doubt, ask!

Summary

What We Covered

  • Accept assignments through GitHub Classroom
  • Clone using GitHub Desktop
  • .gitignore excludes regenerated files (Library folder)
  • .gitattributes configures Git LFS for large binary assets
  • Daily workflow: Pull → Work → Review → Commit → Push
  • Be mindful of storage — don't commit builds or large packages
